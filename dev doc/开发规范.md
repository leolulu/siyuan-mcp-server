# Siyuan MCP 开发规范

## 1. 目标

本规范用于约束 `siyuan-mcp-server` 的日常开发，确保：

- 用户提示可读、可理解。
- 工具行为一致、可维护。
- 后续开发者有统一实现标准，避免重复踩坑。


## 2. 适用范围

- 主要适用于 `src/siyuan_mcp_server/__init__.py` 中所有 `@mcp.tool()` 工具。
- 特别适用于写操作工具：`create_document`、`update_block`、`delete_block`、`insert_block`、`prepend_block`、`append_block`、`move_block`。


## 3. 消息推送规范（重点）

### 3.1 必须统一入口

所有通知统一通过以下函数发送：

- `_push_message(title, content)` - 成功/普通消息
- `_push_error_message(title, content)` - 错误消息

原因：

- 语义化表达，用户一眼看懂发生了什么。
- 避免技术参数堆砌，聚焦实际操作结果。
- 统一兜底逻辑（通知发送失败时的处理）。


### 3.2 文案目标

通知文案必须让普通用户一眼看懂：**实际操作了什么，结果如何**。

避免：

- 技术参数（block_id、markdown_chars、endpoint 等）
- 流程性描述（"第 X 步"、"参数校验通过"）
- 英文调试语句

推荐：

- 语义化描述（"《标题》已保存到 路径"）
- 结果导向（"块内容已更新（+50 字）"）
- 位置清晰（"标题下方添加：新的列表项..."）


### 3.3 通知格式

消息结构：

```
标题：{操作名}
内容：{语义化描述}
```

各操作通知示例：

| 操作 | 标题 | 内容示例 |
|------|------|----------|
| 创建文档 | `创建文档` | `《我的笔记》已保存到 日常/2024-02-27` |
| 更新块 | `更新内容` | `块内容已更新（+50 字）` |
| 删除块 | `删除内容` | `已删除：这是一段被删除的文本...` |
| 插入块 | `插入内容` | `在「参考链接...」之后：新的列表项...` |
| 前置插入 | `前置插入` | `标题「项目计划」开头添加：第一个子项...` |
| 后置插入 | `后置插入` | `标题「项目计划」末尾添加：补充内容...` |
| 移动块 | `移动内容` | `「会议纪要」已移动到「项目计划...」之后（共 5 个块）` |


### 3.4 工具侧实现约定

tool 函数负责：

1. 执行实际操作
2. 提取语义化信息（标题、路径、内容预览、长度变化等）
3. 调用 `_push_message` 或 `_push_error_message`

不再需要：

- 多步骤进度通知
- 技术参数暴露
- 复杂的 stage/detail 结构

## 4. 写操作工具实现规范

写操作工具采用简洁模式：

1. 参数校验
2. 执行操作
3. 发送语义化成功通知
4. 失败时发送错误通知并抛出异常


## 5. Tool Docstring 规范

### 5.1 简单工具

简单只读工具至少包含：

- 一句话说明
- `Args`
- `Returns`


### 5.2 复杂或易误用工具

复杂工具建议补齐以下三段：

- `适用场景`
- `使用方法`
- `注意事项`

特别是存在参数优先级、路径限制、历史依赖、结果体量影响时必须写清楚。


## 6. 块操作语义规范

### 6.1 insert / move 优先级

根据思源 API 语义，涉及锚点时应明确：

- `insertBlock`：`nextID > previousID > parentID`
- `moveBlock`：`previousID` 与 `parentID` 同传时以位置锚点优先


### 6.2 层级稳定性建议

当目标是“确保挂到某个标题（如 H2/H3）下面”时：

- 优先使用 `append_block` / `prepend_block`
- 或只传 `parent_id`

避免将跨层级的 `next_id/previous_id` 与 `parent_id` 混用，导致结果位置与预期不一致。

### 6.3 层级结构移动防呆（新增）

为防止“只移动父块，内容未跟随”造成的结构错位，约定如下：

- `move_block` 必须按“逻辑块组”移动，禁止仅移动父块本身。
- `allow_heading_only_move` 仅保留兼容入参，不再允许绕过防呆（传 `true` 应报错）。
- 调整分节/层级顺序时，必须按“块组”移动：
  - 标题块（h1-h6）：分节范围 = 从标题开始，直到下一个同级或更高级标题（level <= 当前 level）之前的所有块。
  - 其他块：子树块组 = `[目标块, 该块全部后代...]`
  - 执行方式：标题按“分节范围”整体移动（必要时按顶层块链式移动以保持连续）；其他块只移动根块一次（思源会随根块一起移动整棵子树）。
- 每次移动后应重新读取最新结构并重算锚点，禁止基于旧快照连续盲移。
- 若目标是“稳定挂到父块”，优先使用 `append_block` / `prepend_block`，或只传 `parent_id`。

### 6.4 move_block 变更要求（新增）

- 任何改动 `move_block` 行为或参数时，必须同步更新：
  - `src/siyuan_mcp_server/__init__.py` 中 `move_block` docstring
  - `README.md` 的工具说明与安全规程
- 若新增防呆参数，必须给出默认值并保持向后兼容。

### 6.5 文档删除约束（新增）

- MCP 工具禁止提供“删除文档（文档树节点）”能力；该操作仅允许用户在思源客户端手动完成。
- `delete_block` 必须拒绝删除文档块（type='d'），避免误删导致索引异常风险。


## 7. 安全与数据处理规范

- `execute_sql` 仅允许 `SELECT`。
- 历史文件访问路径必须以 `/history` 或 `/data/history` 开头。
- 返回文本中涉及敏感信息时，保持脱敏处理。


## 8. 变更验证规范

代码变更后至少执行：

- `lsp_diagnostics`（检查改动文件）
- `uv run python -m compileall src`
- `uv run python -c "import siyuan_mcp_server; print('ok')"`
- `uv run python -m unittest discover`

若仓库存在历史存量告警/错误，需在提交说明中明确“是否为本次新增”。


## 9. 维护原则

- 新增写操作 tool 时，先复用统一通知链路，不要复制粘贴自定义文案逻辑。
- 需调整文案风格时，优先改 `_notify_progress/_notify_failure` 及其 humanize 映射。
- 文档和实现必须同步更新，避免“代码行为变了，说明没变”。
