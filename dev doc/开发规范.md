# Siyuan MCP 开发规范

## 1. 目标

本规范用于约束 `siyuan-mcp-server` 的日常开发，确保：

- 用户提示可读、可理解。
- 工具行为一致、可维护。
- 后续开发者有统一实现标准，避免重复踩坑。
- **所有写操作必须推送通知，禁止静默执行，确保用户有足够的安全感。**

---

## 2. 核心原则：禁止静默执行

**所有非只读操作（写入、修改、删除、移动等）必须向前台推送通知，严禁静默执行。**

理由：

- 用户需要即时反馈，知道操作是否成功执行。
- 静默执行会让用户失去安全感，无法确认 MCP 是否在工作。
- 通知是用户与 MCP 交互的唯一可见窗口，必须充分利用。

违反此原则的代码在 Code Review 中应被直接驳回。

---

## 3. 适用范围

- 主要适用于 `src/siyuan_mcp_server/__init__.py` 中所有 `@mcp.tool()` 工具。
- 特别适用于写操作工具：创建文档、更新块、删除块、插入块、移动块等。

---

## 4. 消息推送规范（重点）

### 4.1 强制通知要求（最高优先级）

**所有写操作工具必须在执行后推送通知，无例外。**

强制要求：

- ✅ 成功时必须推送成功通知
- ✅ 失败时必须推送错误通知
- ✅ 通知必须在 API 调用完成后立即发送
- ❌ 禁止静默执行（执行了写操作但不推送任何通知）
- ❌ 禁止仅记录日志而不推送前台通知
- ❌ 禁止假设"用户知道发生了什么"而跳过通知

违反此要求的代码视为严重缺陷，必须在 Code Review 中驳回。

### 4.2 必须统一入口

所有通知统一通过内部提供的通知函数发送：

- 成功通知函数 - 用于操作成功反馈
- 错误通知函数 - 用于操作失败反馈

原因：

- 语义化表达，用户一眼看懂发生了什么。
- 避免技术参数堆砌，聚焦实际操作结果。
- 统一兜底逻辑（通知发送失败时的处理）。

### 4.3 文案目标

通知文案必须让普通用户一眼看懂：**实际操作了什么，结果如何**。

避免：

- 技术参数（如内部 ID、字节长度、接口端点等）
- 流程性描述（"第 X 步"、"参数校验通过"）
- 英文调试语句

推荐：

- 语义化描述（说明操作对象和结果）
- 结果导向（说明变更内容和影响）
- 位置清晰（说明操作位置和上下文）

### 4.4 通知格式

消息结构：

```
标题：{操作名}
内容：{语义化描述}
```

各操作通知示例：

| 操作     | 标题       | 内容示例                                            |
| -------- | ---------- | --------------------------------------------------- |
| 创建文档 | `创建文档` | `《我的笔记》已保存到 日常/2024-02-27`              |
| 更新块   | `更新内容` | `「旧内容预览...」→「新内容预览...」（+50 字）`     |
| 删除块   | `删除内容` | `已删除：这是一段被删除的文本...`                   |
| 插入块   | `插入内容` | `在「参考链接...」之后：新的列表项...`              |
| 前置插入 | `前置插入` | `标题「项目计划」开头添加：第一个子项...`           |
| 后置插入 | `后置插入` | `标题「项目计划」末尾添加：补充内容...`             |
| 移动块   | `移动内容` | `「会议纪要」已移动到「项目计划...」之后（共 5 个块）` |

### 4.5 工具侧实现约定

写操作工具函数负责：

1. 执行实际操作
2. 提取语义化信息（如标题、路径、内容预览、变更量等）
3. 调用统一通知函数

不再需要：

- 多步骤进度通知
- 技术参数暴露
- 复杂的中间状态描述

---

## 5. 写操作工具实现原则

写操作工具采用统一的异常处理和通知模式：

标准流程：

1. 参数校验
2. 执行操作
3. 发送语义化成功通知
4. 失败时发送错误通知并抛出异常

所有写操作必须包裹在异常处理结构中，确保：

- 成功时用户收到明确反馈
- 失败时用户知道错误原因
- 异常不会被静默吞掉

---

## 6. Tool Docstring 规范

### 6.1 简单工具

简单只读工具至少包含：

- 一句话说明
- `Args`（参数说明）
- `Returns`（返回值说明）

### 6.2 复杂或易误用工具

复杂工具建议补齐以下三段：

- `适用场景`
- `使用方法`
- `注意事项`

特别是存在参数优先级、路径限制、历史依赖、结果体量影响时必须写清楚。

---

## 7. 块操作语义规范

### 7.1 insert / move 优先级

根据思源 API 语义，涉及锚点时应明确：

- `insertBlock`：`nextID > previousID > parentID`
- `moveBlock`：`previousID` 与 `parentID` 同传时以位置锚点优先

### 7.2 层级稳定性建议

当目标是"确保挂到某个标题（如 H2/H3）下面"时：

- 优先使用 `append_block` / `prepend_block`
- 或只传 `parent_id`

避免将跨层级的 `next_id/previous_id` 与 `parent_id` 混用，导致结果位置与预期不一致。

### 7.3 层级结构移动防呆（新增）

为防止"只移动父块，内容未跟随"造成的结构错位，约定如下：

- `move_block` 必须按"逻辑块组"移动，禁止仅移动父块本身。
- `allow_heading_only_move` 仅保留兼容入参，不再允许绕过防呆（传 `true` 应报错）。
- 调整分节/层级顺序时，必须按"块组"移动：
  - 标题块（h1-h6）：分节范围 = 从标题开始，直到下一个同级或更高级标题（level <= 当前 level）之前的所有块。
  - 其他块：子树块组 = `[目标块，该块全部后代...]`
  - 执行方式：标题按"分节范围"整体移动（必要时按顶层块链式移动以保持连续）；其他块只移动根块一次（思源会随根块一起移动整棵子树）。
- 每次移动后应重新读取最新结构并重算锚点，禁止基于旧快照连续盲移。
- 若目标是"稳定挂到父块"，优先使用 `append_block` / `prepend_block`，或只传 `parent_id`。

### 7.4 move_block 变更要求（新增）

任何改动 `move_block` 行为或参数时，必须同步更新：

- `src/siyuan_mcp_server/__init__.py` 中 `move_block` docstring
- `README.md` 的工具说明与安全规程
- 若新增防呆参数，必须给出默认值并保持向后兼容。

### 7.5 文档删除约束（新增）

- MCP 工具禁止提供"删除文档（文档树节点）"能力；该操作仅允许用户在思源客户端手动完成。
- `delete_block` 必须拒绝删除文档块（type='d'），避免误删导致索引异常风险。

---

## 8. 安全与数据处理规范

- SQL 查询工具仅允许 `SELECT` 语句。
- 历史文件访问路径必须以指定前缀开头。
- 返回文本中涉及敏感信息时，保持脱敏处理。

---

## 9. 变更验证规范

代码变更后至少执行：

- `lsp_diagnostics`（检查改动文件）
- `uv run python -m compileall src`
- `uv run python -c "import siyuan_mcp_server; print('ok')"`
- `uv run python -m unittest discover`

若仓库存在历史存量告警/错误，需在提交说明中明确"是否为本次新增"。
---

## 10. 维护原则

- 新增写操作 tool 时，先复用统一通知链路，不要复制粘贴自定义文案逻辑。
- 需调整文案风格时，优先修改统一通知函数及其辅助函数。
- 文档和实现必须同步更新，避免"代码行为变了，说明没变"。
- **新增或修改写操作 tool 时，必须确保通知链路完整，禁止静默执行。**
